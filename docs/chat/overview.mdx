When you configure an agent, FrogBot stores authenticated HTTP conversations in Payload collections. You can use the default `threads` and `messages` collections without adding configuration, or adopt the roles into your own collections.

Anonymous agent calls remain stateless, even when a custom agent access rule allows them.

## Use the default collections

FrogBot injects both collections when agents are configured and no collection claims either role.

| Collection | Stored fields |
| --- | --- |
| `threads` | `title`, `user`, `agent`, `lastMessageAt` |
| `messages` | Text `id`, `thread`, `role`, `parts`, `metadata`, `usage` |

Thread and message collections use soft deletion and appear under **Chat** in the admin panel. Access is owner-scoped through the thread's `user` relationship.

Messages store the AI SDK `UIMessage.parts` array. Tool calls, tool results, reasoning, and custom data parts therefore remain available on later turns instead of being flattened into text.

## Customize the collections

Add `thread: true` or `message: true` to adopt a collection. FrogBot keeps your slug and merges in its required fields, hooks, access rules, and admin defaults.

```ts
import type { CollectionConfig } from 'frogbot'

export const Conversations: CollectionConfig = {
  slug: 'conversations',
  thread: true,
  fields: [
    {
      name: 'channel',
      type: 'select',
      options: ['web', 'support']
    }
  ]
}

export const Turns: CollectionConfig = {
  slug: 'turns',
  message: true,
  fields: []
}
```

Your field definitions win when they are compatible with the base schema. FrogBot appends missing base fields and combines hooks. It rejects incompatible field types and protects these fields:

- `threads.user`
- `messages.id`
- `messages.thread`
- `messages.parts`

Mark at most one collection for each role. If an unmarked collection already uses the default `threads` or `messages` slug, sanitization asks you to mark or rename it rather than adopting it implicitly.

## Continue a thread

The first authenticated HTTP request creates a thread and returns its ID.

```bash
curl http://localhost:3000/api/agents/support \
  --request POST \
  --header "Authorization: Bearer $FROGBOT_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{ "prompt": "Review campaign performance" }'
```

JSON responses include `threadId`. Stream responses provide the same value in `X-Frogbot-Thread-Id`.

Send that ID with only the next user message. FrogBot loads and validates stored history before running the agent.

```bash
curl http://localhost:3000/api/agents/support \
  --request POST \
  --header "Authorization: Bearer $FROGBOT_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{ "prompt": "Now compare it with last week", "threadId": "THREAD_ID" }'
```

FrogBot rejects a thread owned by another user. It also rejects client-supplied assistant or system messages; only the server persists assistant output.

## Persist local generation

Local generation is stateless unless you provide an existing thread ID.

```ts
const result = await frogbot.agents.support.generate({
  prompt: 'Check the latest campaign changes',
  threadId,
  req,
  overrideAccess: false
})
```

`generate({ threadId })` persists the new user message, loads stored history, runs the agent, and stores the resulting assistant message. Local `stream()` calls remain stateless.

## Understand durability

FrogBot awaits the user-message write before inference begins. Streaming responses save finalized or partial assistant parts on completion, error, abort, or client disconnect. Usage is attached when the model emits it but never blocks the message save.

<Warning>
  Payload's SQLite adapter disables transactions unless you configure `transactionOptions`. Without transactions, a failure between thread creation and the first message write can leave an empty thread. MongoDB and transaction-enabled adapters roll back the pair.
</Warning>

A hard process exit during inference can still leave a user message without an assistant reply. The thread remains valid and can be continued; incremental part persistence and resumable streams are planned separately.
